Before You Start

Prerequisites

You need three things installed on your computer. That's it. Everything else is handled by the SDK and HYTOPIA's platform.

Required

Node.js 18 or higher

The JavaScript runtime that runs your game server. Download from nodejs.org. After install, run node --version in your terminal — you should see v18.x.x or higher.

Required

Git

Version control. Download from git-scm.com. After install run git --version. You'll also need a free GitHub account at github.com to host the repo.

Required

VS Code (recommended)

A free code editor from Microsoft. Download from code.visualstudio.com. Install the TypeScript + JavaScript extension for syntax highlighting and autocomplete on the SDK.

Optional but helpful

HYTOPIA Account

Create a free account at hytopia.com. You'll test your game at hytopia.com/play by connecting to your local server. No account required for local dev, but needed to publish.

💡
Terminal / Command Line

Every command in this guide is run in your terminal (Mac: Terminal.app, Windows: PowerShell or Windows Terminal). When you see a line starting with $, that's a terminal command — type everything after the $ and hit Enter.

Code Organization

Project Structure

Set up this folder structure before writing any game code. Each folder has one job. When you ask an LLM to write code, you can tell it exactly which file to create.

blimp-game/ index.ts ← entry point. only wires things together. package.json tsconfig.json .gitignore README.md CONTRIBUTING.md LICENSE │ src/ ← all game logic. one concern per file. playerState.ts ← player data, points, inventory, tier partsData.ts ← part definitions, costs, variants tierLogic.ts ← tier unlock checks & effects airship.ts ← airship entity, flight physics, controls resourceBlocks.ts ← block placement, harvest, respawn islands.ts ← upper island detection & discovery events workshop.ts ← NPC entity & shop interaction constants.ts ← all magic numbers in one place │ assets/ map/ world.json ← exported from build.hytopia.com ui/ index.html ← HUD overlay (points, parts, tier) shop.html ← parts shop panel audio/ wind-ambient.mp3 discovery.mp3 tier-unlock.mp3 models/ ← gltf models if any
🗂️
Why separate src/ files?

Keeping each system in its own file means: (1) when you ask an LLM to fix a bug, you only paste the one relevant file, (2) when the game grows you know exactly where to look, (3) collaborators can work on different files without conflicts. index.ts should only import and connect things — never contain game logic directly.

Create these folders now. They can be empty — just having the structure in place keeps you organized as you build each phase.

terminal
$ mkdir -p src assets/map assets/ui assets/audio assets/models
Version Control

Git Workflow

You don't need complex branching for a solo prototype. One simple rule: commit after every working phase. Never commit broken code to main.

Branch Strategy

One branch per phase

Create a branch before starting each build phase. When the phase works and is tested, merge it to main. This means main is always a working game.

# Start a new phase $ git checkout -b phase/1-world-setup # When done and tested $ git checkout main $ git merge phase/1-world-setup $ git push
Commit Message Style

Conventional commits

Every commit message starts with a type prefix. This makes your git history readable like a changelog and helps future contributors (and you) understand what changed.

featadd resource blocks with 30s respawn on starting island
featimplement build points system and player state map
featadd HUD overlay with points counter and parts grid
fixresource block respawn not clearing on player leave
choreadd constants.ts with all tunable values
docsupdate README with current phase progress
stylealign HUD colors with Blimp forest/amber palette
⚠️
What to .gitignore

Your repo was created with the Node .gitignore template, which already excludes node_modules/. Also make sure .env is ignored if you ever add API keys (for future AI agent features). Never commit secrets to a public repo.

AI-Assisted Development

Working with an LLM Effectively

Since you're not a coder, the LLM is your developer. These rules make the difference between getting clean, working code and getting a mess you can't fix.

1

Always give context first

At the start of every new conversation, paste:

  • Your current index.ts
  • The specific src/ file you want to modify
  • A one-line description: "This is a HYTOPIA SDK voxel game called Blimp."

Without this, the LLM guesses. With it, the LLM writes code that fits your existing architecture.

2

One phase at a time

Never ask for the whole game at once. Each phase prompt should say: "Do not add X or Y yet — only implement Z." This keeps the output focused and testable. If you get broken code, it's one small thing to fix, not a 500-line mess.

3

Ask for explanations

After receiving code, ask: "Explain what each function does in plain English." This builds your understanding and catches logic errors before you run anything. You don't need to understand TypeScript — you need to understand the game logic.

4

Test before committing

Run the server (npm run dev), open HYTOPIA play, and verify the specific thing you just built works. Only then run git commit. If it breaks, paste the error message to the LLM and ask it to fix that specific error — don't ask it to rewrite everything.

5

Keep a prompt log

Create a file called PROMPTS.md in your repo. After each phase, paste the prompt you used and note what worked or what you had to adjust. This becomes an invaluable reference when you return to the project or want to onboard a contributor.

PROMPTS.md
✦ Master prompt template — paste this at the start of every phase conversation
CONTEXT: I'm building a browser game called Blimp using the HYTOPIA SDK (TypeScript/JavaScript). Blimp is a solarpunk airship builder: players earn Build Points, buy airship parts, assemble their ship, and unlock flight tiers. TECH STACK: - HYTOPIA SDK (npm package: hytopia) - TypeScript - Server-authoritative (all game logic runs on server) - Player UI = HTML overlay files in assets/ui/ PROJECT STRUCTURE: index.ts → entry point only src/playerState.ts → PlayerState type and Map src/partsData.ts → part definitions src/tierLogic.ts → tier unlock logic src/airship.ts → airship entity + flight src/resourceBlocks.ts → resource block system src/workshop.ts → NPC + shop src/constants.ts → all tunable values src/islands.ts → upper island discovery CURRENT CODE: [paste your current index.ts here] [paste the specific src/ file you're modifying here] THIS PHASE TASK: [describe exactly what to build — see each phase below] RULES: - Do not add anything beyond what's in THIS PHASE TASK - Write clean, commented TypeScript - Use the HYTOPIA SDK API only (import from 'hytopia') - Tell me which file each code block belongs to

Phase Zero

Environment Setup & Boilerplate

Install the SDK, run the boilerplate, confirm you can connect to a working game server in your browser. Don't write any Blimp code yet.

1

Initialize the npm project

In your blimp-game directory:

terminal
$ npm init -y $ npm install hytopia $ npm install --save-dev typescript @types/node
2

Add TypeScript config

Create tsconfig.json in the root:

tsconfig.json
{ "compilerOptions": { "target": "ES2020", "module": "ESNext", "moduleResolution": "bundler", "strict": true, "outDir": "./dist", "rootDir": "." }, "include": ["index.ts", "src/**/*"] }
3

Add dev script to package.json

Open package.json and add a dev script:

package.json (scripts section)
"scripts": { "dev": "npx hytopia" }
4

Copy the boilerplate index.ts

From the SDK you downloaded, copy boilerplate/index.ts into your root. Also copy boilerplate/assets/map.json into assets/map/world.json. This is your baseline.

index.tsassets/map/world.json
5

Run it and verify

terminal
$ npm run dev

You should see Server running on port 8080. Open your browser and go to: https://hytopia.com/play?connect=127.0.0.1:8080. You should be able to walk around in the default world. If this works — you're ready.

6

First commit

terminal
$ git add . $ git commit -m "chore: init project with HYTOPIA SDK boilerplate" $ git push

Phase One

Sky World & Resource Blocks

Replace the default flat world with the Blimp sky map. Add glowing resource blocks that players harvest for Build Points. Set the visual mood.

Create a branch first: git checkout -b phase/1-world-setup

1

Build the world map

Go to build.hytopia.com. Build three layers: a starting island at Y=0–20, open sky above, and 3 small island platforms at Y=90. Keep it simple — blocky, a bit mossy, solarpunk. Export as JSON and save to assets/map/world.json.

💡
LLM can't build the map

The visual world map must be built in the HYTOPIA world editor — it's a GUI tool. The LLM can place individual blocks programmatically in code (which is how resource blocks and trigger zones work), but the terrain base map is your job in the editor.

2

Create constants.ts — all tunable values in one file

This is a discipline that pays off enormously. Every number that might need tuning lives here. You never have to hunt through code to change the point value of a block.

src/constants.ts
// All tunable game values in one place. // Change numbers here — never hunt through game files. export const POINTS = { RESOURCE_BLOCK_HARVEST: 10, RESOURCE_BLOCK_RESPAWN_MS: 30_000, // 30 seconds SOLAR_SAIL_PASSIVE_PER_SEC: 1, ISLAND_DISCOVERY_BONUS: 50, }; export const TIERS = { T1_REQUIRED_PARTS: ['envelope', 'frame'], T2_REQUIRED_PARTS: ['gondola', 'propulsion'], T3_REQUIRED_PARTS: ['navigation', 'lifesystems'], }; export const FLIGHT = { T1_LIFT_SPEED: 4, T1_DRIFT_FORCE: 0.5, T2_TURBINE_SPEED: 8, T2_SOLAR_SPEED: 5, T3_MAX_ALTITUDE: 200, T2_MAX_ALTITUDE: 60, }; export const WORLD = { SPAWN: { x: 0, y: 5, z: 0 }, RESOURCE_BLOCK_COUNT: 20, ISLAND_Y: 90, };
src/constants.ts
3

✦ Phase 1 LLM Prompt

Phase Four

Assembly Bench & Tier 1 Flight

The first magical moment. Player assembles Envelope + Frame, the airship entity spawns, and for the first time they rise into the sky.

git checkout -b phase/4-tier1-flight

1

LLM prompt: tier logic + assembly bench

✦ Phase 4a LLM Prompt
PHASE 4a TASK — Tier Logic & Assembly Create src/tierLogic.ts. Export function: checkAndUnlockTier(world, player, playerState): - If tier is 0 and player has 'envelope' and 'frame' in inventory → unlock Tier 1 - If tier is 1 and player has 'gondola' and 'propulsion' → unlock Tier 2 - If tier is 2 and player has 'navigation' and 'lifesystems' → unlock Tier 3 - On unlock: update playerState.tier, send STATE_UPDATE to HUD, play tier-unlock audio, send congratulations chat message Assembly Bench: - Add a second interactable Entity (Assembly Bench) on the starting island at {x: -5, y: 1, z: 0} - On interact: send UI data { type: 'BENCH_OPEN', inventory, tier } - When player confirms assembly (UI sends 'ASSEMBLE'), call checkAndUnlockTier - If Tier 1 just unlocked, call spawnAirship(world, player) (to be implemented next step)
src/tierLogic.tsindex.ts
2

LLM prompt: airship entity + Tier 1 flight

✦ Phase 4b LLM Prompt
PHASE 4b TASK — Airship Entity (Tier 1) Create src/airship.ts. spawnAirship(world, player, tier, propulsionVariant?): - Spawn a custom Entity with kinematic RigidBody above the player's current position (+5Y) - The entity represents the airship (use a simple voxel block structure for now) - Attach the player to it: on each world TICK, set player entity position to ship position + gondola offset - Store { shipEntity, tier, propulsionVariant } in a Map keyed by playerId Tier 1 flight controls (per tick): - Read player input (PlayerInput from world tick) - SPACE → apply upward velocity (FLIGHT.T1_LIFT_SPEED from constants) - SHIFT → apply downward velocity - No horizontal movement yet - Apply random small XZ drift each tick to simulate wind (FLIGHT.T1_DRIFT_FORCE) - Clamp Y to max FLIGHT.T2_MAX_ALTITUDE (same ceiling applies at T1) Do not add steering yet — Tier 1 is vertical-only + wind drift.
src/airship.ts

Phase Five

Tier 2 — Cruiser Steering

Add WASD directional controls to the airship. Propulsion variant determines move speed. Solar Sails grant a passive points bonus.

git checkout -b phase/5-tier2-steering

✦ Phase 5 LLM Prompt
PHASE 5 TASK — Tier 2 Steering Update src/airship.ts to add Tier 2 behavior. When player is at Tier 2, per tick: - W → apply forward velocity in the direction the player camera faces (XZ plane) - S → apply backward velocity - A / D → strafe left/right - SPACE / SHIFT still handle vertical - Speed based on propulsionVariant: - 'turbines' → FLIGHT.T2_TURBINE_SPEED - 'solar_sails' → FLIGHT.T2_SOLAR_SPEED - Remove the wind drift from Tier 1 (player is in control now) - Altitude ceiling stays at FLIGHT.T2_MAX_ALTITUDE Solar Sails passive bonus: - If propulsionVariant is 'solar_sails' and player is airborne - Every second, call addPoints(playerId, POINTS.SOLAR_SAIL_PASSIVE_PER_SEC) - Use a per-player timer, not a global interval Also update index.ts: - After Tier 2 unlock: send chat message explaining new controls - Update HUD STATE_UPDATE to include propulsionVariant

The win moment

When a player first reaches an upper island — that's the prototype's "wow." Test this personally. Fly up slowly, hit the island, hear the discovery sound. If it doesn't feel satisfying, ask the LLM to add a camera shake impulse and a particle burst at the moment of discovery.

Phase Seven

Polish, Tune & Ship

The game mechanics are done. Now make it feel good. Tune progression pacing, add atmosphere, write the onboarding flow, and prepare for public release.

git checkout -b phase/7-polish

1

Full playthrough tune

Play the full game from spawn to Tier 3 island discovery. Tune values in constants.ts so: Tier 1 takes ~5 min, Tier 2 ~10 min, Tier 3 ~15 min. Never edit raw numbers in game files — only in constants.ts.

2

Welcome sequence

On JOINED_WORLD, send a timed sequence of chat messages that explain the loop:

in index.ts — welcome sequence
setTimeout(() => chatManager.sendPlayerMessage(player, '⎈ Welcome to Blimp.', '7ed957'), 1000); setTimeout(() => chatManager.sendPlayerMessage(player, 'Collect glowing green blocks to earn Build Points.'), 3000); setTimeout(() => chatManager.sendPlayerMessage(player, 'Visit the Workshop (glowing NPC) to buy airship parts.'), 5500); setTimeout(() => chatManager.sendPlayerMessage(player, 'Assemble your ship at the Assembly Bench to take flight.'), 8000);
3

Atmosphere pass

Ask the LLM to add: particle emitters at wind current positions in the sky layer, ambient bird/wind audio that increases in volume as the player ascends, and a subtle fog layer at high altitude (Tier 3 zone). Each of these is a small independent addition — add them one at a time and test each.
